# Реестр багов

**Обновлено:** 2026-04-17 (BUG-0008)  
**Формат:** один баг — один блок, разделены `---`

## Процесс

| Статус | Значение |
|---|---|
| `новый` | Обнаружен, репро не подтверждён |
| `в работе` | Патч разрабатывается |
| `исправлен` | Патч применён, в рантайме не проверен |
| `подтверждён` | Проверено в живом потоке |
| `закрыт` | Нет воспроизведений в последующих сессиях |
| `отменён` | Фича удалена или проблема снята |

**Приоритеты:**
- P0 — плеер умер/завис, краш-луп, потеря данных
- P1 — крупный UX-слом, автоплей или управление ненадёжны
- P2 — неверный дисплей/состояние, но есть обходной путь
- P3 — косметика или редкая проблема

**Области:** autoplay, queue, playback, ui, metrics, infra, prefetch, db

## BUG-0001

**Статус:** исправлен  
**Приоритет:** P1  
**Область:** autoplay  
**Первый раз:** 2026-04-15T10:08:03Z  
**Последний раз:** 2026-04-15T10:14:10Z

**Суть:** Режим ∞ зацикливается на одном треке после цикла автоплея.

**Шаги воспроизведения:**
- Включить автоплей (∞) в активной гильдии
- Дать поиграть треку «Emancipator "Baralku"» в автоплей-потоке
- Около конца трека spawn автоплея возвращает `skip_recent_dup`
- Idle-fallback через навигацию перезапускает тот же URL

**Ожидалось:** Следующий трек — другой.  
**Факт:** Один и тот же URL воспроизводится несколько раз подряд.

**Доказательства:**
- `data/metrics/session-tracks.txt` — тот же videoId два раза подряд (`source=navigation`)
- `data/metrics/autoplay-spawn.txt` — `outcome=skip_recent_dup`
- `data/metrics/skips.txt` — `quick_skip source=navigation` на том же URL

**Исправление:** Защита от self-loop в `skip()` в `src/music.js` через `sameYoutubeContent()`. Неверный/исчерпанный `idleBackForwardTail` очищается, не ставится в очередь.

## BUG-0003

**Статус:** исправлен  
**Приоритет:** P1  
**Область:** autoplay  
**Первый раз:** 2026-04-17T00:10:00Z  
**Последний раз:** 2026-04-17T00:18:00Z

**Суть:** Spawn автоплея занимал 25–27 секунд → «Ошибка взаимодействия» при нажатии кнопок.

**Шаги воспроизведения:**
- Включить ∞, скипнуть трек при пустой очереди
- Наблюдать состояние загрузки 25–30 секунд
- Нажать любую кнопку панели во время загрузки

**Ожидалось:** Следующий трек найден за 5–8 секунд.  
**Факт:** Spawn занял 25–27 с. Discord interaction timeout = 15 с после `deferUpdate` → «Ошибка взаимодействия».

**Доказательства:**
- Лог: `[autoplay] spawn end guild=... elapsed=27503ms`
- Groq сам быстрый (650–860 мс); всё время — в поиске
- `runAutoplaySearchStep` делал `Promise.all` по 2–3 запросам Groq × 5 yt-dlp = до 15 процессов одновременно при семафоре = 3

**Причина:** Параллельный поиск создавал очередь из 15 yt-dlp в семафоре на 3 слота → ~5 последовательных раундов по ~4 с = 20+ с. При этом `pickFirstNonEmptyResultSet` всё равно использовал только первый непустой результат — 80% поиска тратилось впустую.

**Исправление:**
- `autoplay-spawn.js` → `runAutoplaySearchStep` (в Шаге 9 inline'ен как private helper, ранее в `autoplay-spawn-search.js`): `Promise.all` заменён на sequential `for...of` с ранним выходом (стоп после первого непустого результата)
- `youtube-search.js` → `MAX_CONCURRENT_YTDLP_SEARCH`: дефолт 3 → 5

**Результат:** 27 с → 4.8–8 с (улучшение в 3–5 раз). Ошибки взаимодействия устранены.

## BUG-0004

**Статус:** исправлен  
**Приоритет:** P0  
**Область:** prefetch  
**Первый раз:** 2026-04-17T00:21:00Z  
**Последний раз:** 2026-04-17T00:21:00Z

**Суть:** `ReferenceError: Cannot access 'guardSpawnStale' before initialization` при pool_hit.

**Шаги воспроизведения:**
- Включить ∞; дать хотя бы одному автоплей-треку поиграть (чтобы full-фаза prefetch успела)
- Скипнуть — pool_hit срабатывает, вызывается prefetch fast path
- Бот крашится с `unhandledRejection`, следующий трек не играет

**Ожидалось:** Pool hit → следующий трек за ~50 мс.  
**Факт:** `ReferenceError: Cannot access 'guardSpawnStale' before initialization` в `music.js:484`. Воспроизведение останавливается, гильдия зависает в LOADING.

**Причина:** `guardSpawnStale` объявлен через `const` в конце `_spawnAutoplayPlaylistImpl` (~строка 557). Но вызывался в prefetch fast path (~строка 484) до объявления — TDZ ошибка. Баг был латентным: prefetch pool никогда не заполнялся из-за BUG-0003 (spawn был настолько медленным, что пользователь успевал скипнуть раньше). Исправление BUG-0003 позволило pool_hit сработать — и BUG-0004 проявился.

**Исправление:** Удалён вызов `guardSpawnStale(...)` из prefetch fast path. `popBestCandidate` уже валидирует `sessionId` и `generation` до возврата кандидата — избыточная проверка не нужна.

**Результат:** Pool hit завершается за ~5 мс без ошибок.

## BUG-0006

**Статус:** исправлен  
**Приоритет:** P1  
**Область:** autoplay, prefetch  
**Первый раз:** 2026-04-17  
**Последний раз:** 2026-04-17

**Суть:** Трек повторяется — после естественного завершения тот же трек играет снова.

**Шаги воспроизведения:**
- Включить ∞, дать нескольким трекам поиграть до конца (не скипать)
- Наблюдать: один из уже сыгранных треков играет повторно

**Ожидалось:** Автоплей выбирает новый трек.  
**Факт:** Тот же трек добавляется в очередь снова и играет повторно.

**Причина (два бага в связке):**

1. **`candidate-ranker.js` создаёт spread-копии**: `rankAutoplayCandidates` делает `const item = { ...raw }` для каждого кандидата — возвращает **новые объекты**, а не исходные ссылки из `items`.

2. **Surplus фильтр в `music.js` использовал reference equality**: `items.filter((it) => it !== pickedForNotify ...)`. Поскольку `pickedForNotify` — spread-копия, а `items` содержит оригиналы, `it !== pickedForNotify` **всегда true** для всех элементов. Весь массив `items` попадал в surplus, включая только что поставленный в очередь трек.

3. **Prefetch pop path не вызывал `isRecentBlocked`**: Когда surplus-трек поднимался из пула, в prefetch-пути не было проверки недавно сыгранных треков — только `pushQueueIfNotQueued` (проверяет лишь что трека нет в текущей очереди). Поскольку трек уже играет (не в очереди), он проходил и добавлялся снова.

4. **`FULL_SEARCH_LIMIT` не объявлена** в `autoplay-prefetch-runner.js` — вызывало `ReferenceError` в full-фазе prefetch, фаза молча падала через catch.

**Исправление:**
- `music.js`: surplus фильтр → URL-сравнение + `isYoutubeUrlBlockedForAutoplaySpawns` фильтр:
  ```javascript
  const pickedUrl = pickedForNotify?.url ?? null;
  const surplus = items.filter(
    (it) => it?.url && it?.title && it.url !== pickedUrl && !isYoutubeUrlBlockedForAutoplaySpawns(id, it.url),
  );
  ```
- `music.js`: prefetch pop path → добавлена проверка `isYoutubeUrlBlockedForAutoplaySpawns` перед `pushQueueIfNotQueued`
- `autoplay-prefetch-runner.js`: добавлена константа `FULL_SEARCH_LIMIT = 8`

## BUG-0007

**Статус:** исправлен  
**Приоритет:** P3  
**Область:** ui  
**Первый раз:** 2026-04-17  
**Последний раз:** 2026-04-17

**Суть:** Никнейм пользователя отображается в тексте панели рядом с названием трека.

**Шаги воспроизведения:**
- Добавить трек через `/музыка` или кнопку `+ добавить`
- Дождаться начала воспроизведения
- Посмотреть на текст в панели с кнопками

**Ожидалось:** `Сейчас играет: **Aimer - Brave Shine**`  
**Факт:** `Сейчас играет: **Aimer - Brave Shine** · Soupsoup` — никнейм добавившего человека виден в панели.

**Причина:** В `music-panel.js` функция `_panelContent` формировала строку с `requestedByFrag = info?.requestedByName ? \` · ${info.requestedByName}\` : ''`. Этот фрагмент добавлялся в текст панели, хотя никнейм уже отображается в строке сессионного списка.

**Исправление:** Удалён `requestedByFrag` из строки `_panelContent`. Никнейм по-прежнему виден в списке треков (сессионное сообщение), но убран из панели.

## BUG-0008

**Статус:** исправлен  
**Приоритет:** P2  
**Область:** ui, autoplay  
**Первый раз:** 2026-04-17  
**Последний раз:** 2026-04-17

**Суть:** При выключении ∞ во время поиска следующего трека панель зависает в нерелевантном состоянии («Сейчас ничего не воспроизводится» или «∞ Подбираем…») вместо нормального idle.

**Шаги воспроизведения:**
- Включить ∞ (автоплей)
- Дождаться когда трек закончится и бот начнёт подбирать следующий (LOADING / "∞ Подбираем следующий трек…")
- Пока идёт поиск — нажать ∞ чтобы выключить автоплей

**Ожидалось:** Панель корректно показывает `Сейчас ничего не воспроизводится.` Кнопка ∞ не подсвечена.  
**Факт:** Панель показывала «∞ Подбираем следующий трек…» при выключенном автоплее, либо зависала в непоследовательном состоянии. При быстром повторном нажатии ∞ возможна ситуация когда панель показывает idle хотя трек ещё играл.

**Причина (три слоя):**

1. **`music.js` / `runPlayNext`**: после того как spawn завершается с `skip` (stale guard убил его из-за смены генерации), код всегда ставил `IDLE_EXHAUSTED + AUTOPLAY_ERROR` независимо от того, включён ли сейчас автоплей. В итоге состояние «ищем трек» оставалось даже когда автоплей уже выключен.

2. **`music-panel.js` / `_panelStatusLine`**: отображение `PANEL_AUTOPLAY_WAIT = '∞ Подбираем следующий трек…'` происходило при `statusReason === AUTOPLAY_ERROR` без проверки — включён ли автоплей в данный момент. Если ∞ выключили, а reason не сбросился — панель продолжала показывать текст поиска.

3. **`music.js` / `toggleAutoplay`**: при выключении автоплея не вызывался `notifyPlaybackUiRefresh`, поэтому панель не перерисовывалась немедленно при смене состояния кнопки ∞.

**Исправление:**
- `music.js` / `runPlayNext`: после завершения spawn проверяется `autoplayByGuild.has(id)`. Если автоплей уже выключен → `setPlayerState(IDLE_EXHAUSTED, StatusReason.NONE)`, без `AUTOPLAY_ERROR`.
- `music-panel.js` / `_panelStatusLine`: `PANEL_AUTOPLAY_WAIT` показывается только при `statusReason === AUTOPLAY_ERROR` И `getMusicTransportState(guildId)?.autoplay === true`. Иначе — `PANEL_IDLE`.
- `music.js` / `toggleAutoplay`: добавлен вызов `notifyPlaybackUiRefresh(id)` при выключении, чтобы панель немедленно обновилась.
