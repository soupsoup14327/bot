## Мотивация

### Исходная проблема (до шагов 0–10)

Изначально `music.js` раздувался до ~1800+ строк и держал внутри `Map<guildId, GuildMusicState>` — «мешок» разнородных полей (`queue`, `player`, `connection`, `playing`, `ytdlp`, `ffmpegNormalize`), к которым напрямую обращались десятки функций. Итеративные вынесения (`idle-navigation-*`, `autoplay-*`, `stop-and-leave-steps.js`) не устраняли корневую связность.

Типичные проявления до декомпозиции:

- `BUG-0008` (панель гаснет после toggleAutoplay) — возник повторно из-за гонки нескольких асинхронных писателей в один UI-state.
- `Bug #2` (кнопки гаснут при skip) — транзитное состояние `s.playing=false` между треками не явное, читается как «ничего не играет».
- `Bug #4` (repeat+skip → тот же трек) — порядок операций `queue.shift` / `player.stop` / `repeatByGuild.delete` не детерминирован.
- `Bug #5` (eternal loading на третьем треке) — `enqueue` на 30-секундном `entersState` таймауте, при частичном fail'е оставляет inconsistent state между `s.connection` и `s.player`.
- Stderr listener одновременно отвечает за playability cache (консервативный порог) и за repeat force-skip (реактивный порог) — склеен в один shared mutable flag.

### Текущее состояние (после шага 10)

`GuildMusicState` и цикл воспроизведения живут в `playback-loop.js`; `music.js` — thin facade (~670 строк) с target'ами для `orchestrator.commands.*` и без владения per-guild Map очереди/процессов. Оставшийся осознанный компромисс — толстый `enqueue` и часть UI-запросов в том же файле; вынесение в `enqueue-pipeline.js` зафиксировано как опциональный шаг 10b.

### Что этот рефакторинг решает

1. **Растворяет `GuildMusicState` «мешок».** Каждый модуль владеет своим per-guild state.
2. **Вводит явный state machine стрима (`StreamHandle`)** — phase lifecycle видимый и тестируемый.
3. **Чёткое разделение `commands.*` / `events.*`** в orchestrator'е — внешние клиенты (Discord, HTTP, Web) больше не импортируют `music.js` напрямую.
4. **Атомарные операции voice lifecycle** в `voice-adapter`.
5. **Заготовка для мульти-источника** (`audio-pipeline` как boundary) без его немедленного внедрения.
6. **Unit-покрытие чистой критической логики** — чтобы следующий рефакторинг не делался «с чистого листа».

## Ключевые контракты

### `StreamHandle`

Владелец lifecycle одной попытки воспроизведения одного ресурса.

```
StreamHandle {
  resource:     AudioResource          // для player-controller
  handles:      { ytdlp, ffmpeg }      // для teardown
  meta: {
    provider:     'youtube' | ...      // multi-source ready
    resolvedUrl:  string
    durationSec:  number | null
    hasNormalize: boolean
    startedAt:    number               // Date.now() на момент create
  }
  whenStable:   Promise<void>          // resolves когда phase достиг STABLE
  whenEnded:    Promise<EndReason>     // resolves при TERMINATED

  cancel():     void                   // orchestrator вызывает на skip/stop/nav
  notifyPlayerState(status): void      // player-controller уведомляет о transitions
}
```

**Phase state machine** (внутри StreamHandle):

```
SPAWNED    — процесс запущен, player ещё не в Playing
FLOWING    — player впервые вошёл в Playing
STABLE     — player держал Playing T секунд суммарно, без захода в AutoPaused/Idle
ENDING     — cancel() вызван, или process выходит
TERMINATED — все процессы закрыты, whenEnded resolved
```

**Stability timer:** T = 2.5s (первая версия, подстраиваем по данным shadow-режима). User Paused не сбрасывает накопленный STABLE, AutoPaused — сбрасывает.

### `EndReason` (возвращается из `whenEnded`)

```
EndReason =
  | { kind: 'natural' }                 // exit 0, стрим доиграл
  | { kind: 'cancelled' }               // cancel() вызван (skip/stop/nav)
  | { kind: 'fatal', class: FatalClass, context: string }
  | { kind: 'transient', context: string }

FatalClass =
  | 'region_blocked'     // HTTP 403, geo-restriction
  | 'video_unavailable'  // removed, private, deleted
  | 'age_restricted'     // sign in to confirm
  | 'embed_disabled'     // playback on other sites disabled
  | 'extractor_error'    // yt-dlp не справился (возможно устарел)
  | 'network_error'      // pre-FLOWING сетевая проблема
  | 'unknown_fatal'      // non-zero exit без match'а allowlist
```

### `stream-error-classifier.js`

**Pure function**, без внутреннего state.

```
classify({
  source:      'ytdlp_stderr' | 'ffmpeg_stderr' | 'process_exit' | 'process_error',
  line:        string | null,
  phase:       'SPAWNED' | 'FLOWING' | 'STABLE' | 'ENDING' | 'TERMINATED',
  processCode: number | null,
  signal:      string | null,
}) → { severity: 'fatal' | 'transient' | 'warning' | 'unknown',
       fatalClass: FatalClass | null,
       reason: string }
```

**Fatal allowlist** (по `line`, не по `includes('error')`):

- `HTTP Error 403` / `404` / `410` → provider verdict
- `Video unavailable` / `This video is not available` / `Video has been removed` / `Private video` → video removed
- `Sign in to confirm your age` → age-gated
- `Playback on other websites has been disabled` → embed blocked
- `Unable to extract` / `ERROR: Unsupported URL` → extractor issue

**Transport noise** (Broken pipe, Invalid argument, EPIPE, ECONNRESET) — **не в allowlist**. Если прилетело в phase ≥ STABLE — классифицируется как `transient`. До STABLE + non-zero exit → `unknown_fatal`.

### Playability cache policy (confidence → TTL)

| Сценарий | Cache write | TTL |
|---|---|---|
| provider verdict + phase < FLOWING | confident bad URL | длинный (часы) |
| provider verdict + phase < STABLE | bad URL | средний |
| provider verdict + phase ≥ STABLE | possibly bad | короткий (минуты) |
| transport noise + phase < STABLE | bad URL | короткий |
| transport noise + phase ≥ STABLE | — | нет записи |
| cancelled | — | нет записи |

### `orchestrator` API

```
orchestrator.commands.*             (use-case API — Discord, HTTP, WS)
  enqueue(guildId, input) → Result
  skip(guildId, actor) → Result
  pause(guildId) → Result
  resume(guildId) → Result
  previousTrack(guildId, actor) → Result
  toggleRepeat(guildId) → Result
  toggleAutoplay(guildId) → Result
  stopAndLeave(guildId) → Result

orchestrator.events.*               (внутренние реакторы, void)
  onVoiceReady(guildId, channelId)
  onVoiceGone(guildId, reason)
  onPlayerIdle(guildId, meta)
  onStreamError(guildId, endReason)
  onAutoplaySpawned(guildId, items)
```

**Mutex-friendly design.** `commands.*` проектируются так, как будто вызываются через per-guild Promise-chain (хотя на первом этапе реальный mutex не внедрён — event loop сериализует синхронные команды).

### Новый тип сигнала

- **`track_stream_failed`** — SessionSignal (best_effort), эмитится при `EndReason.kind === 'fatal'` до того как `schedulePlayNext` запускает следующий трек. Recommender учится различать user-dislike (track_skipped с актором) и technical failure. Не влияет на personal preferences.

## Правила владения состоянием

| State | Owner (write) | Readers |
|---|---|---|
| `QueueItem[]` | queue-manager | orchestrator, HTTP API |
| `AudioPlayer` instance | player-controller | — (инкапсулировано) |
| `playing` flag | player-controller | orchestrator, transport-state |
| `VoiceConnection` | voice-adapter | — (инкапсулировано) |
| `ytdlp` / `ffmpeg` процессы | StreamHandle / audio-pipeline | — (per-stream) |
| `phase` lifecycle | StreamHandle | — (наружу `whenStable` / `whenEnded`) |
| `repeat` / `autoplay` флаги | guild-session-state (storage), orchestrator (policy) | транспорт, API |
| `currentPlaying*` | orchestrator (после успешного старта) | UI, autoplay, navigation, signals |
| `sessionId`, `botVoiceState` | voice-adapter (events) → orchestrator (policy) → guild-session-state (storage) | API, signals |
| `PlayerState` (UI FSM) | orchestrator | UI, API, WS |
| autoplay session context | autoplay-session-state (через autoplay-spawn) | autoplay-spawn |
| prefetch pool | autoplay-prefetch (через autoplay-spawn, prefetch-runner) | autoplay-spawn |

**Одно правило:** каждый state имеет **одного owner** (модуль с mutation API) и **любое число readers** (через read API).

### Шаг 10 — `playback-loop.js` ✅ ВЫПОЛНЕН

**Цель:** довести `music.js` до facade-состояния, вынеся ядро воспроизведения в отдельный модуль.

**Audit `music.js` до шага — 1233 строки, 4 роли:**

| Блок | Строк | Что внутри |
|---|---|---|
| Callback wiring | ~140 | setOnPlaybackUiRefresh, setOnAutoplaySpawned, setOnMusicForceStop, setOnPlayingTrackDisplay, setOnPlaybackIdle + инициализация spawner'а |

**Выполненное действие — extraction `playback-loop.js`:**

Перенесено в новый `src/playback-loop.js` (~683 строки):
- `guildState: Map<string, GuildMusicState>` + `GuildMusicState` typedef (per-guild proc tracker).
- Процесс lifecycle: `killYtdlp`.
- Player event handlers: `handlePlayerIdle`, `handlePlayerError`, `handlePlayerStateChange` (+ регистрация через `registerPlayerControllerCallbacks` при импорте).
- Playback цикл: `runPlayNext`, `schedulePlayNext` (через `createSchedulePlayNext`), `dequeueQueueItemAndScheduleNext`.
- Streaming шаг: `streamUrl` (audio-pipeline → procs → StreamHandle → history/metrics записи).

**Публичный контракт `playback-loop.js`:**
- `ensureGuildMusicState(id)` — get-or-create per-guild state (раньше local `getState`).
- `getGuildMusicState(id)` — read-only accessor без создания (для `skip`/`previousTrack`/`stopAndLeave`).

- `schedulePlayNext(id, reason)` — сериализованная версия `runPlayNext` (используется из `enqueue`, `previousTrack`).

**Сохранённая обратная совместимость:**
`music.js` реэкспортирует все 4 setter'а — клиенты (`index.js`, `music-panel.js`) не меняются.

**Регрессия попутно:** при запуске после Шага 7 выяснилось, что `src/index.js:119` использовал прямой вызов `stopAndLeave`, но функция не была импортирована (Шаг 7 перевёл всё на `orchestrator.commands.*`). Заменено на вызов через оркестратор. В `verify-modules.mjs` добавлен smoke-тест `playback-loop public API` для ловли похожих регрессий ранее.

**Изменённые файлы:**
- `src/playback-loop.js` — новый, 683 строки.
- `src/music.js` — **1233 → 640 строк** (-48%). Удалены 459 строк тела извлечённых функций + ~20 импортов, ставших мёртвыми (`AudioPlayerStatus` — кроме `getMusicTransportState`, `createSchedulePlayNext`, `createStream`, baseline/metrics импорты, используемые только в runPlayNext/streamUrl, и т.д.).

- `scripts/verify-modules.mjs` — +1 smoke group `playback-loop public API` (exports + idempotent killYtdlp + null-safe getGuildMusicState).

**Что НЕ вынесено (осознанно):**
- `enqueue` (130 стр) — остался в `music.js`. Функция уже одна, её ответственности (voice connect / playability check / queue push / title resolve / metrics log) уже делегированы соответствующим модулям. ### Шаг 10b — `enqueue-pipeline.js` ⏸ ОТЛОЖЕН (идея)

**Статус:** записано на будущее, не выполнять без явного запроса.

**Идея:** вынести `enqueue` (130 строк в `music.js`) в отдельный модуль `enqueue-pipeline.js`. Функция — самая сложная из оставшихся в facade'е: `ensureVoiceConnection` → normalize URL → playability hard-skip check → queue push (+ reset idle cursor/tail, очистка сессии если плеер простаивает) → `schedulePlayNext` на первый трек → title resolve для panel hint → `logQueueAdd`.

**Почему отложено:**
- `enqueue` уже **одна функция** — все её ответственности уже делегированы соответствующим модулям (`voice-adapter`, `queue-manager`, `playability-cache`, `youtube-search`, `playback-loop`, `playback-metrics`). Вынос — косметический, без уменьшения сложности.
- Extraction потребует либо передачи ~10 deps через DI (шумно), либо сохранения прямых импортов в новом модуле (тогда просто движение строк).
- `music.js` после Шага 10 был ~640 строк; сейчас ~670 (мелкий рост после шага — см. таблицу модулей в `АРХИТЕКТУРА.md`) — читаемость facade'а не страдает.

**Когда делать:**
- Если появится второй consumer для этого pipeline (например, batch-enqueue из API для плейлистов, который не хочет возвращать `panelHint`).
- Если `enqueue` начнёт расти (вторая voice-connection ветка, дополнительные источники трекa).

**Возможный дизайн, если делать:**
```js
// enqueue-pipeline.js
export async function runEnqueuePipeline({ channel, queryOrUrl, source, userId, userDisplayName }) {
  // 1. ensureVoiceConnection + ensureGuildMusicState
  // 2. normalize + hard-skip check
  // 3. push to queue + reset session if idle
  // 4. schedulePlayNext if first item
  // 5. resolve title for panelHint (lazy)
  // 6. logQueueAdd
  return { panelHint, trackLabel };
}
```
`music.js::enqueue` становится 3-строчной обёрткой над этим pipeline.

| Модуль | Тип | Причина |
|---|---|---|

### Не blocker на первом этапе

- `player-controller.js` — stateful, но через DI mock-friendly
- `voice-adapter.js` — stateful, нужен мок `@discordjs/voice`
- `orchestrator.js` — integration, не юнит
- End-to-end с реальным ботом

**Правило:** каждый stateful модуль проектируется так, чтобы его **можно было** покрыть интеграционно потом. То есть зависимости через DI, а не прямой импорт. Тесты не писать сейчас — но архитектурно держать дверь открытой.

## Операционные решения

### Test runner

**`node:test`** (встроенный). Причины:
- Node ≥ 18, ESM — работает без конфигурации.
- Ноль новых runtime/dev зависимостей.
- Минимальный инфраструктурный шум для unit-тестов на чистую логику.

### Документация

- **`АРХИТЕКТУРА.md`** — актуальная структура модулей, потоки и границы; обновляется при смене кода.
- **`ПЛАН-РЕФАКТОРИНГА.md`** — журнал шагов 0–10 и контракты; не дублирует архитектуру, но сохраняет обоснования и «почему так».
- **`СТАТУС.md`** — что закрыто, что в обкатке, что отложено (в т.ч. HTTP API, PG ops).

### Rollback policy

- **Шаги 0-4:** чистый `git revert` PR'а. Риск минимален (pure код или shadow).
- **Шаги 5-7:** каждый Шаг проходит минимум один день на проде перед merge следующего. Regression → immediate revert.
- **Шаг 8-9:** как 5-7, но риск ниже (autoplay изолирован от основного playback).

### Shadow observation — логи

- **Шаг 1 classifier:** `bot/data/metrics/classifier-shadow.txt`.
- **Шаг 2 StreamHandle:** `bot/data/metrics/stream-handle-shadow.txt`.
- Формат — как у существующих метрик (см. `НАБЛЮДАЕМОСТЬ.md`): одна строка на событие, TSV / лёгкий JSON.
- Флаг `METRICS_TXT_ENABLED` контролирует запись, как для остальных метрик.

### Feature flags

На первом этапе отдельные `ENV` флаги не нужны. Shadow mode (Шаги 1-2) включается на всём проекте через `METRICS_TXT_ENABLED=1`. Переключение на new path (Шаг 6) — атомарно, без флага (в случае regression — revert PR'а).

## Что НЕ входит в этот рефакторинг

Эти пункты **обсуждались** и **включены в архитектуру** (контракты готовы), но **реализация отложена**:

1. **HTTP API / WebSocket для Web / Android клиентов.** Код в `src/api/*` есть, к `index.js` не подключён; `commands.*` готовы принять HTTP-клиентов. См. [СТАТУС.md](СТАТУС.md).
2. **Плейлисты и расширенная персистентность.** Лайки и playback history уже в БД (ADR-001); отдельные плейлисты в БД и эволюция `music-signals` в полноценный event bus — вне завершённых шагов 0–10.
3. **Multi-source** (Spotify, локальные файлы). `audio-pipeline` спроектирован как abstraction point, но реализуется только `youtube` provider.
4. **Per-guild command mutex.** `commands.*` проектируются mutex-friendly, но реальный mutex внедряется только когда появится async-вызов из двух клиентов одновременно (HTTP + Discord).
5. **Reconnect policy для voice.** Текущее поведение «упал коннект — кончилась сессия» сохраняется. Reconnect — отдельная задача после Шага 5.
6. **Consumer event bus для `music-signals`.** Сейчас подписчики (autoplay-context, recommendation-bridge) хардкодом. Формальный pub/sub — отдельная эволюция после стабилизации текущего слоя.
