## 1. Планировщик воспроизведения (`schedulePlayNext`)

### Инварианты

1. **`schedulePlayNext` никогда не `await`-ится изнутри `runPlayNext`**, пока текущий `runPlayNext` ещё «держит» job в цепочке (иначе deadlock: job = `prev.then(runPlayNext)`, `prev` никогда не завершится). После `await spawnAutoplayPlaylist` — только **`void schedulePlayNext(id, 'autoplay-after-spawn')`**.

2. **Дедуп головы:** перед `unshift` в ветке `[prev/idle]` и при `skip` с `idleBackForwardTail` — `unshiftQueueIfNewHead`. Если голова — уже тот же ролик — не дублировать. Всё равно сделать **`void schedulePlayNext(..., 'prev-idle-dedup-kick')`** при необходимости «пнуть» цепочку.

3. **Автоплей push:** `pushQueueIfNotQueued`. Если после цикла `added === 0` → `return 'skip'` (ничего не ставим в очередь, не трогать `schedulePlayNext`).

4. **Idle → `schedulePlayNext(id, 'player-idle')`:** только при `wasPlaying`; иначе двойной Idle съест очередь.

### Кто вызывает `schedulePlayNext`

| reason | Откуда |
|---|---|
| `player-idle` | `AudioPlayerStatus.Idle` после `wasPlaying` |
| `autoplay-after-spawn` | После успешного `spawnAutoplayPlaylist` и непустой очереди |
| `play-error-recovery` | `catch` в `runPlayNext` |
| `playability-hard-skip` | Известный bad URL — снять с очереди без стрима |
| `enqueue-first` | `enqueue`, первый трек в пустой очереди |
| `prev-idle` / `prev-idle-dedup-kick` | `previousTrack`, ветка `endedAll` |

## 2. Prefetch-пулы

### Структура

Два логических пула на гильдию в `autoplay-prefetch.js`:
- `guildId:fast` — результат фазы 1 (быстрой, без Groq)
- `guildId:full` — результат фазы 2 (полной, с Groq)

Каждый пул хранит: `{ phase, sessionId, generation, candidates[] }`

### Правила записи

**`storeFast`:**
- Пишет только если нет подходящего `guildId:full` с той же версией
- Не бампает `generation`

**`storeFull`:**
1. Проверить: `sessionId` и `generation` не устарели
2. Проверить: в `candidates` есть валидные элементы
3. **Только после этого** атомарно бампнуть `generation` и записать `guildId:full`
4. `guildId:fast` становится stale автоматически (generation не совпадает)

> Важно: generation бампается только при успешной записи валидного full-пула, **не** при старте фазы.

### Правила чтения

`popBestCandidate(guildId, { sessionId, generation })`:
1. Взять из `guildId:full`, если версия совпадает
2. Иначе взять из `guildId:fast`, если версия совпадает
3. Иначе вернуть `null` → inline spawn

### Правила инвалидации

| Событие | Действие |
|---|---|
| `quick_skip` | Удалить оба пула, бампнуть generation |
| `skip` | Pop head из `full`, если есть; иначе pop из `fast` |
| `stop` / `voice_disconnect` / `session_reset` / `autoplay_off` / `user_enqueue` | Удалить оба пула, бампнуть generation |

### Контракт фазы 1 (fast)

- 1 дешёвый semantic seed (нормализованный тайтл)
- 1 прямой yt-dlp поиск
- 1–2 кандидата
- Rank/filter обязателен (quality guards как в полном spawn)
- Никакого fanout, никакого Groq

## 3. Stale guard (поколения spawn)

`autoplay-stale-guard.js` хранит `spawnGeneration` по гильдии.

Когда **бампается** generation:
- `stopAndLeave`
- Выход из войса
- Выключение ∞
- `user_enqueue` (новый трек от пользователя)
- `quick_skip`
- Успешная запись full-пула (`storeFull`)

Каждый async spawn захватывает generation в начале. После retrieval, после поиска, перед enqueue — сверяется с текущим. Несовпадение → тихий выход (`return 'stale'`).

**Нельзя** вызывать `guardSpawnStale` до его объявления (`const`) в том же scope — TDZ ошибка (см. BUG-0004).

## 4. Дедуп очереди

`queue-invariants.js`:
- `pushQueueIfNotQueued(queue, item)` — не добавлять, если тот же `videoId` уже в хвосте
- `unshiftQueueIfNewHead(queue, item)` — не вставлять в голову, если голова — тот же `videoId`
- Сравнение через `sameYoutubeContent(a, b)` → `extractYoutubeVideoId`

**Нет дублей по source:** один и тот же URL может быть добавлен дважды от разных `source` только если `videoId` различается. По `videoId` — строгий дедуп.

## 5. PlayerState и StatusReason

Единственная точка записи — `guild-session-state.setPlayerState(guildId, state, reason?)`.

| PlayerState | Когда |
|---|---|
| `IDLE` | Нет активного воспроизведения |
| `LOADING` | Идёт `runPlayNext` / spawn |
| `PLAYING` | `player.play()` вызван, AudioPlayer активен |
| `PAUSED` | Пользователь поставил на паузу |
| `ERROR` | Критический сбой (stream, yt-dlp) |

| StatusReason | Когда ставить |
|---|---|
| `NONE` | Нормальная смена состояния |
| `AUTOPLAY_DISABLED` | Стоп без новых треков и ∞ выключен |
| `VOICE_ERROR` | Сбой голосового соединения |
| `TRACK_BLOCKED` | hard-skip заблокированного URL |
| `STREAM_FAIL` | yt-dlp / ffmpeg ошибка |

> **Долг:** `VOICE_ERROR` и `TRACK_BLOCKED` объявлены, но `music.js` их не выставляет. Требует доработки.

## 6. Навигация (idle/prev)

FSM в `idle-navigation-state-machine.js`:
- Чистая логика, без side effects
- Входные данные: текущее состояние + действие (`prev`, `idle`, `skip`, `clear`)
- Выходные данные: новый курсор, URL для воспроизведения, инструкция для очереди

Ломать только через `idle-navigation-machine-api.js` → `applyIdleNavigationTransition`.  
`idle-navigation-apply.js` применяет результат FSM к реальной очереди.

**Не** обращаться к `pastTrackUrls` / `idleNavCursor` напрямую из `music.js` — только через API модуля.
